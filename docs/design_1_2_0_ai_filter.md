# 1.2.0 AI 필터 설계안

## 목표
- AI 필터 기능 부활 및 개선
- 온디바이스 대신 클라우드형 LLM 기반으로 방향 전환
- 실사용 가능한 UX 확보
- 기존 rule-based 필터 뒤의 2차 보조 엔진으로 운영
- 배치 검사 + 후속 차단 구조로 효율화

## 핵심 원칙
1. 기존 rule-based 필터는 1차, AI 필터는 2차 보조 엔진
2. AI 입력은 항상 원문 전체 기준
3. 게시글과 댓글 판정 결과는 반드시 분리
4. AI는 판정 엔진, 앱은 최종 집행 엔진
5. 실패 시 자동 차단 금지
6. 배치 검사를 기본으로 하되, 대형 글도 검사 생략 없이 단독 전체 검사

## 입력 원칙
- 배치 대상이든 단독 대상이든 모두 원문 전체를 AI에 전달
- 포함 정보:
  - 게시글 번호
  - 제목 전체
  - 작성자 ID/IP
  - 작성자 닉네임
  - 본문 전체
  - 본문 미디어 소스 전체
  - 댓글 전체
- 댓글은 반드시 후속 매핑 가능한 식별자(commentId 또는 index)를 포함
- 요약/축약/발췌 금지

## 배치 검사 구조
- 일반 글: 배치 큐에 적재 후 일괄 검사
- 대형 글: 단독 검사 큐로 보내되 원문 전체를 그대로 검사
- 배치 발사 조건은 용량 중심 + 시간/개수 보조
  - 누적 용량 도달
  - 최대 글 수 도달
  - 최대 대기 시간 초과
- 용량 기준은 입력 축약 기준이 아니라, 몇 개를 한 번에 묶을지 정하는 기준

## AI 출력 구조
- 게시글과 댓글 결과를 반드시 분리
- 권장 스키마 예시:

```json
{
  "results": [
    {
      "post_no": 123,
      "post_decision": "pass",
      "post_reason": "",
      "comments": [
        {
          "comment_id": "c1",
          "decision": "block",
          "reason": "광고 댓글"
        }
      ]
    }
  ]
}
```

- 게시글 decision: pass / review / block
- 댓글 decision: pass / review / block
- reason 필드 포함

## 후속 처리 구조
- 게시글 block → 게시글 차단 큐
- 댓글 block → 댓글 차단 큐
- review → 자동 차단하지 않고 로그/보류 처리
- 집행 직전 대상 재검증 권장

## Provider 방향
- 클라우드형 provider 구조 유지/확장
  - OpenAI-compatible
  - Gemini direct
- 봇별 설정:
  - AI 필터 사용 여부
  - provider
  - endpoint
  - api key
  - model
  - 사용자 프롬프트
  - review-only 여부
  - 자동 차단 허용 여부
  - 배치 최대 글 수
  - 배치 최대 대기 시간
  - 배치 최대 용량 기준

## 활동 로그 방향
- AI 배치 처리 상태를 활동 로그에 명확히 표시
- 예시:
  - [AI 배치] 후보 5건 수집
  - [AI 배치] 검사 요청 시작
  - [AI 배치] 완료 / post 1건 block / comment 3건 block / review 2건
  - [AI 게시글 차단] ...
  - [AI 댓글 차단] ...
  - [AI 오류] ...

## 실패 처리 원칙
- 아래 경우 자동 차단 금지:
  - 응답 파싱 실패
  - decision 누락
  - 매핑 실패
  - timeout
  - 429
  - 5xx
  - provider 오류
- 실패는 review 또는 로그 처리
- 일부 결과 누락 시 누락 대상은 차단 금지
